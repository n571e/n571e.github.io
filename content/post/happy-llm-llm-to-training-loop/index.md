把视角从 Transformer 的模块细节再往上提一层之后，我越来越清楚地感觉到：**大语言模型把语言模型底座、分阶段训练和一整套输入输出协议连成了一个系统。**

真正让我把这件事想通的，是几条原本分开的线第一次在脑子里接上了：一条是训练目标、能力边界和对齐思路，另一条是 tokenizer、样本构造、loss mask、generate 这些具体实现。本文想做的，就是把这些关键知识点重新整理成一条更顺的主线。

---

## 一、先把 LLM 的整体框架理顺

在看这些内容之前，我对“大语言模型”的理解其实还是有点笼统：知道它更大、更强，却不太能清楚说明“强”到底强在哪里。真正需要理顺的，恰恰是几个最基础的问题：它的底层目标是什么，能力为什么会变，训练为什么要分阶段。

### 1.1 LLM 的本质仍然是语言模型

首先需要校正的一个认知是：**LLM 的底层目标并没有神秘化，它本质上仍然是在做 next-token prediction。**

如果写成概率形式，就是：

$$
p(x) = \prod_{t=1}^{T} p(x_t \mid x_{1:t-1})
$$

这意味着，无论一个模型后来表现得多像“助手”、多像“推理器”、多像“知识库”，它在训练底座上依然是在学习“给定前文，下一个 token 最可能是什么”。这一点非常关键，因为它会直接影响后面我对 Pretrain、SFT 和生成过程的理解。很多看起来很高阶的能力，最后都还是从这个基本目标一路长出来的。

### 1.2 LLM 的强，来自规模化系统共同作用

另一个很重要的点是：**不要把 LLM 简化成“更大的 BERT”或者“参数更多的 Transformer”。**

模型能力的变化，背后往往是多种因素一起放大的结果：

- 模型规模变大
- 数据规模变大
- 训练算力变大
- 训练策略和数据配比更成熟

也正因为如此，LLM 的能力提升不能只理解成“网络更深了，所以更聪明了”。更准确的说法是：**规模化训练让一个原本只会做语言建模的系统，在某些阈值上开始表现出新的能力形态。**

### 1.3 我终于把“涌现、上下文学习、指令遵循、逐步推理”分开了

以前看这些词，经常会觉得它们都在描述“模型变强了”，但越往下看越会意识到，这些其实对应着不同层面的能力。

- **涌现能力** 说的是：当模型、数据和训练规模跨过某个阈值之后，一些原来很弱甚至几乎看不见的能力会突然变得明显。
- **上下文学习（In-Context Learning）** 说的是：模型不需要参数更新，只通过 prompt 和 few-shot 示例，就能在当前上下文里临时适配一个任务。
- **指令遵循** 说的是：模型开始学会把输入理解成“用户要求我做什么”，并按要求组织回答。
- **逐步推理** 说的是：模型在处理复杂问题时，能够通过中间步骤把问题拆开，而不是一次性拍出最终答案。

把这四者分开之后，我对 LLM 的理解就更稳定了。因为它让我知道，当我看到一个模型“好像很聪明”的时候，应该继续追问：它到底是知识更充分、上下文利用更好、指令理解更稳，还是推理链条更完整？

### 1.4 多语言、长上下文、多模态是扩展能力，幻觉是系统风险

还有一个需要尽早建立的判断是：**能力扩展和可靠性不是同一件事。**

模型可以支持多语言、能看更长上下文、甚至能接入图像等多模态输入，但这并不自动意味着它“更可信”。相反，幻觉这件事提醒我，大语言模型本质上依然是概率生成系统。它的输出可能很流畅、很像专家，但这和“它一定知道自己在说什么”不是一回事。

这一点对学习者非常重要，因为它决定了我以后看待模型输出时的态度。长上下文不等于稳定推理，助手风格也不等于事实可靠，语言流畅性更不等于知识正确性。

### 1.5 Pretrain、SFT、RLHF/DPO 之所以要分开，是因为它们在解决不同问题

最能搭起框架感的一点，是要把大模型训练拆成几个不同阶段来理解。我以前当然知道这些术语，但真正按“它们分别在解决什么问题”去看时，整个训练链条才开始清楚。

| 阶段 | 核心目标 | 我学到的关键理解 |
|------|----------|------------------|
| **Pretrain** | 学知识底座与文本分布 | 让模型先变成一个会建模语言分布的系统 |
| **SFT** | 把模型塑形成能回答问题的助手 | 重点是教会它输入输出格式、回答风格和对话模式 |

这个拆分特别重要，因为它直接改变了我对大模型训练的理解。“训练一个 LLM”包含几种完全不同的目标：Pretrain、SFT 和偏好对齐各自解决不同问题。只有把这些阶段分开理解，后面看代码、看数据格式、看 loss 的时候才不会混。

## 二、再把最小训练闭环看清楚

如果前面解决的是“为什么”，那么接下来需要回答的就是“这些东西到底会在代码和流程里落在哪里”。

在真正把训练流程串起来之前，我对“从零搭一个 LLM”这件事多少带着点敬畏，好像它对应的是一整套遥不可及的工程系统。但至少在理解层面上，我们完全可以先抓住一条更小却很清楚的主线：

> **tokenizer -> 数据样本构造 -> Decoder-Only 模型 -> loss 计算 -> 训练循环 -> 生成**

只要这条链条打通了，很多以前看起来散乱的概念就会开始自动连起来。

### 2.1 Tokenizer 是一整套输入协议

这也是我对 tokenizer 理解变化最大的一点。以前我更多还是把它当成“把文本切成 token”的工具，后来才慢慢意识到，它其实是一种**输入协议**。

比如，在 BPE、ByteLevel、NFKC 这些技术选择之外，更重要的其实是：

- 特殊 token 如何定义
- 对话起止符号如何设计
- 角色边界如何标记
- 编码和解码是否能稳定保真
- chat template 如何把多轮对话拼成单个序列

一旦把 tokenizer 理解成“协议”，很多后续问题就变得容易理解了。模型为什么能知道现在是 user 在说话还是 assistant 在说话？为什么它知道哪里该开始回答？为什么多轮对话还能维持结构？这些都来自 tokenizer 和模板对输入世界的规定。

### 2.2 Decoder-Only 模型的关键，不在于堆模块，而在于理解每个设计在解决什么问题

看到 LLaMA 风格的模型实现之后，我也重新理解了“看模型结构图”的方式。以前我很容易把这些模块记成名词列表，现在我更愿意追问：**它为什么会在这里，它解决的是什么问题？**

先看整体结构，会更容易理解后面那些看似分散的设计选择其实是在同一条主干上协同工作的：

<div align="center">
  <img src="llama2-architecture.png" alt="LLaMA2 模型结构" width="100%" />
  <p>图 3. LLaMA2 结构图，图片引自 Happy-LLM 原文</p>
</div>

- `RMSNorm` 让我意识到，归一化的目标是更稳定地训练深层网络。
- `RoPE` 让我开始从“相对位置建模”的角度理解位置编码，而不是只记住一种公式。
- `GQA` 和 `repeat_kv` 让我看到注意力结构也可以围绕显存与效率做取舍。
- `SwiGLU` 风格的 MLP 让我意识到前馈网络会直接影响表达能力。
- `embedding` 和 `lm_head` 权重共享，则把“输入词表”和“输出词表”这两件事重新连在了一起。

换句话说，重要的是开始从“设计意图”理解模型结构。

### 2.3 Pretrain 和 SFT 的底座其实没有变，真正变化的是监督边界

这一点大概是我最想记住的知识点之一。

无论是预训练数据，还是 SFT 数据，最底层都还是在做自回归建模。也就是说，数据往往还是会被整理成类似这样的形式：

- `X = input_ids[:-1]`
- `Y = input_ids[1:]`

也正因为如此，我第一次真正理解了：**Pretrain 和 SFT 共享同一个 CLM 底座。**

它们真正的差别，主要出现在“哪些 token 参与监督”这件事上。预训练通常会让整个序列都参与语言建模，而 SFT 则更强调：上下文要喂给模型看，但只有 assistant 的回答部分才应该承担 loss。

也就是在这里，`loss_mask` 这个实现细节一下子变得非常有意义。它把“只塑形回答行为”这个抽象目标，准确地落到了训练信号的边界上。

### 2.4 generate 让我真正理解了大模型为什么是“一个 token 一个 token 地长出来的”

看 `generate` 方法时，我第一次真正把推理过程想顺了。

生成时，模型会在当前上下文上做前向传播，然后只取最后一个位置的 logits，再根据 `temperature`、`top_k` 等采样策略决定下一个 token，接着把这个新 token 拼回上下文，继续下一轮循环。

这个过程让我第一次很直观地理解到：

- 为什么训练和推理虽然相关，但运行方式并不完全一样
- 为什么采样参数会显著影响模型输出风格
- 为什么大模型输出会一点点长出来

理解了这个闭环之后，很多以前显得神秘的现象就变简单了。比如为什么模型有时会重复、为什么有时会越说越跑偏、为什么不同采样策略会导致完全不同的结果，这些都和这个逐 token 的生成循环直接相关。

## 三、把概念和实现接起来之后，LLM 才真正清楚

最有价值的地方，是原本分开的东西终于在脑子里接上了。

一边是在解释 **LLM 为什么要这样训练**，另一边是在展示 **这些训练目标会如何落到具体的数据和代码结构里**。

一旦把两边接起来，很多原来只是“知道名词”的内容，就会变成真正能解释系统的知识。

### 3.1 从 next-token prediction 到 X/Y shift

先理解 `next-token prediction`，再去看 `X/Y shift`，很多东西就一下子通了。这个抽象目标在代码里并不玄学，它会非常直接地变成：

- 输入看前面的 token
- 标签预测后一个 token

也就是最基本的 `shift` 操作。

这时我才真正明白，所谓“语言模型底座”是一种会直接决定数据组织方式和 loss 定义的训练协议。

### 3.2 从“助手塑形”到 loss_mask

把 SFT 理解成“助手塑形”之后，还得继续追问：这个目标在实现里到底是怎么落下来的？真正具体起来，它最后会收缩成一个非常明确的问题：

> 哪些 token 应该参与监督，哪些 token 只是上下文条件？

也正因为这样，`loss_mask` 浓缩了整个 SFT 思路。用户输入和历史对话决定模型该如何理解问题；真正参与监督的是 assistant 的回答部分。这个边界一旦清楚，我对 SFT 的理解也就不再停留在概念口号上了。

### 3.3 从“多轮对话能力”到 special tokens 与 chat template

多轮对话能力也一样。只说它是在 SFT 和对齐阶段被塑造出来，还不够具体；还得补上后半句：训练数据必须先被组织成模型能理解的样子。

这时 `special tokens` 和 `chat_template` 的意义就变得很清楚了。它们在告诉模型：

- 一轮对话从哪里开始
- user 和 assistant 的边界在哪里
- 多轮历史应该怎样拼接
- 哪一段是当前应该回答的部分

从这个角度看，对话能力是一套训练协议、数据格式和监督方式共同塑造出来的行为结果。

### 3.4 从“长上下文与位置感”到 RoPE

长上下文同样如此。只说它是 LLM 的一种重要扩展能力还不够，因为它背后牵涉到位置编码、训练长度、推理缓存和注意力开销。

`RoPE` 让我第一次更具体地理解到：模型要处理更长的上下文，本质上还是要解决“位置信息如何进入注意力计算”这个问题。也就是说，长上下文能力不是一句抽象宣传语，它背后对应的是非常具体的位置编码设计。

### 3.5 从“训练阶段”到“训练闭环”

训练阶段的拆分，最后也会落实成一条可执行的训练闭环。一个能跑起来的训练系统，至少要把很多基础环节连起来：

- 学习率调度
- warmup
- 梯度累积
- AMP 混合精度
- 梯度裁剪
- checkpoint 保存

虽然我现在关注的重点不是工程细节本身，但这些内容还是帮我建立了一个很重要的感觉：**训练从来不是只写一个 loss 就结束了。**  
模型目标、数据格式、mask 设计、训练循环、生成行为，这些环节其实共同构成了一个完整闭环。缺了其中任何一块，理解都会变得片面。

## 四、如果只保留最重要的知识点，我会记住这几条

学到这里，我觉得自己最想留下来的是下面这些足够稳定的认知。

### 4.1 LLM 先是语言模型，然后才是助手

它的根始终在 `next-token prediction`。只有先接受这一点，后面关于对齐、助手风格、推理链、工具调用之类的能力，才有一个稳定的起点。

### 4.2 LLM 的能力要拆开看，不能混成一句“它更聪明了”

涌现、上下文学习、指令遵循、逐步推理，其实对应着不同类型的能力变化。把这些概念分开，是理解模型行为边界的前提。

### 4.3 三阶段训练对应的是三类不同问题

Pretrain 学的是语言和知识底座，SFT 学的是助手式回答行为，RLHF/DPO 学的是偏好与安全边界。它们对应不同目标。

### 4.4 很多“能力”最后都会落成训练协议

对话能力会落到 `special tokens` 和 `chat_template`，SFT 会落到 `loss_mask`，语言建模目标会落到 `X/Y shift`，长上下文能力会落到 `RoPE`。当我开始这样看问题时，抽象概念就不再漂浮。

### 4.5 理解 LLM，最重要的是把概念、数据和代码三层接起来

只学概念，容易空；只看代码，容易碎；只有把“为什么这样设计”和“它在实现里长什么样”连起来，知识才会真正变成自己的。

## 五、写在最后：我现在会怎样理解 LLM

这些内容最后帮我建立起来的，是这样一个更完整的大模型认知框架：

> **LLM = 概率语言模型底座 + 分阶段训练 + 一整套输入输出协议。**

这个框架对我来说非常重要，因为它让我以后再看一个模型、一个训练脚本，或者一种新的对齐方法时，会自然地追问：

- 它的底层目标是什么？
- 它属于训练链条的哪一个阶段？
- 它改变的是知识、行为、偏好，还是输入输出协议？
- 它在实现里最终落到了什么地方？

我觉得最重要的收获也就在这里。它让我第一次能把“大语言模型为什么这样训练”和“这些训练为什么会这样写”连成一条线。

接下来如果继续往后学 LoRA、QLoRA，或者进一步看 DPO、对齐训练，我希望自己还能继续沿着这条线往下走。因为一旦主线清楚了，后面的新知识就不再像一堆散落概念，而更像是在同一张地图上不断补细节。
